Code Review — RabbitMQ Reliability

Repository: qmsg · Branch: review/rabbitmq-reliability · Date: 2026-07-16
Focus: long-running service resilience — the service must reconnect and retry RabbitMQ connections indefinitely.

Summary

The framework has a sound recovery design on paper: it leans on the RabbitMQ Java client's automatic connection recovery and backs it with its own scheduled start-retry loop. In practice, the retry loop contained a bug that limited it to a single retry attempt ever, and the fallback was skipped in exactly the situations where automatic recovery cannot help (channel-level errors). Combined, a broker outage at startup, a broker error that closes the channel, or an abandoned recovery would leave the service permanently stuck in starting with no consumer — silently, for the rest of the process lifetime. The publisher side opened a new TCP connection per message and did not use publisher confirms.

All eight findings below marked FIXED are implemented on this branch and the build passes (mvn clean install -Dmaven.test.skip=true). Items marked RECOMMENDATION are deliberate scope decisions left for a follow-up.

Findings and fixes

1. Start retry fires at most once, ever · Critical FIXED

BatchReceiver.scheduleStartRetry()

The guard returned early whenever retryExecutor already existed: 

if (retryExecutor != null && !retryExecutor.isShutdown()) {
  return;   // executor exists after the FIRST retry and is never shut down between retries
}

The executor is created on the first schedule and only torn down at application shutdown (shutdownRetrier()). So after the first retry attempt fired and failed, every subsequent scheduleStartRetry() call returned without scheduling anything. Concrete scenario: broker down at boot → attemptStart() fails → one retry after 5 s → still down → no further retries → service stuck in starting until restarted, even when the broker comes back a minute later.

Fix: the condition now reuses the executor and only creates it when missing or shut down; the task is scheduled unconditionally afterwards. The retry loop now continues indefinitely (every 5 s) until it succeeds or the service is stopped.

2. No reconnect path for channel-level failures · Critical FIXED

BatchReceiver.unavailable()

if (!(connection instanceof Recoverable && channel instanceof Recoverable)) {
  scheduleStartRetry();   // skipped whenever automatic recovery is enabled — i.e. always
}

With automatic recovery enabled (the default), the fallback retry was never scheduled — the code trusted automatic recovery completely. But the RabbitMQ Java client's automatic recovery only covers connection-level failures. A channel closed by a broker error (e.g. 404 after a queue is deleted, 406 precondition failure) is never auto-recovered: the connection stays healthy, recovery never starts, and the consumer is dead. The shutdown listener fired, set the state to starting, and then nothing ever happened again — a silently dead long-running service.

Fix: unavailable() now always schedules the fallback retry. If automatic recovery wins the race, recovered() cancels the pending retry, and the new guard in attemptStart() (finding 3) makes a late-firing retry a no-op. The retry path itself handles a dead channel on a live connection correctly: receiver.open() sees isOpen() == false, closes the remnants, reconnects, redeclares the queue and re-registers the consumer.

3. Duplicate-consumer race between retry and auto-recovery · Critical FIXED

AbstractBatchService.attemptStart()

attemptStart() unconditionally called receiver.consume(...). If a scheduled retry fired after automatic recovery had already restored the consumer (topology recovery re-registers consumers under the same explicit consumer tag queue.<Service>.consumer), the second basicConsume with an already-used tag would fail with a channel error — killing the very channel that had just recovered. With finding 2 making fallback retries routine, this race had to be closed in the same change.

Fix: attemptStart() returns early when the service is already started with a live consumer (consumerTag != null && receiver.isOpen()), and recovered() additionally cancels any pending retry.

4. Unreadable message causes infinite hot redelivery loop · Critical FIXED

BatchReceiver.handleDelivery() · AbstractBatchService.getReader()

getReader() catches deserialization failures and returns null; handleDelivery() then invoked processor.process(null), which threw a NullPointerException, which was caught and answered with basicNack(requeue=true). With basicQos(1) the same broken message is redelivered immediately and exclusively: an infinite tight loop that blocks the entire queue, burns a CPU core and floods the logs — triggered by a single malformed JSON payload sent to the queue.

Fix: a null message (unreadable — it can never succeed on redelivery) is now rejected with basicNack(requeue=false) and logged at error level. Note this drops the message unless a dead-letter exchange is configured — see finding 9, which is the proper home for such messages.

5. Emitter: new connection per message, no publisher confirms · High FIXED

BatchEmitter.ServiceEmitter.emit()

Every emit(...) executed rabbitMQClient.connect() — a full TCP + AMQP handshake — published one message and tore the connection down. Under load this is a connection churn storm (broker connection-rate limits, file-descriptor pressure, latency). Worse for reliability: messages were published with deliveryMode=2 (persistent) but without publisher confirms, so a broker failure at the wrong moment loses the message with no error reported to the caller — the durable delivery mode alone does not guarantee the broker ever received it.

Fix: ServiceEmitter now caches one connection + channel, lazily (re)created and health-checked per publish. The channel is put in confirm mode (confirmSelect()) and each publish waits for the broker's confirm (waitForConfirmsOrDie, 30 s timeout). On any publish failure the emitter closes the stale resources, reconnects and retries once; a second failure propagates to the caller (the REST endpoint returns an error — the caller must know the message was not accepted). The emitter is closed on application shutdown via the service's ShutdownEvent observer.

6. stop()/shutdown aborts mid-way on a broken channel · High FIXED

BatchReceiver.cancel() · AbstractBatchService.stop() / onApplicationShutdown()

cancel() wrapped basicCancel failures in UncheckedIOException. A broken channel during stop() therefore aborted the method after the state was set to stopping — receiver.close() never ran, the state never reached stopped, and in the shutdown observer shutdownRetrier() was skipped, leaking the retry executor thread.

Fix: cancel() is now best-effort (log + continue) — it is only used on the stop/shutdown path, which must always reach close(). The shutdown observer wraps stop() in try/finally so the retrier and the emitter are always closed.

7. ack/nack failure escapes the delivery callback · Medium FIXED

BatchReceiver.handleDelivery()

If the channel died between receiving a delivery and acknowledging it, basicAck/basicNack threw IOException out of the consumer callback into the client library's exception handler, losing context. Fix: the ack/nack is wrapped and logged with queue and delivery tag; the unacked message is redelivered after recovery, and the shutdown listeners already drive the reconnect.

8. Emitted messages have no id · Medium FIXED

BatchEmitter.ServiceEmitter.emit(Message, Serializer)

Message.id was never populated on emit, so the consumer-side BatchContext.traceId (and the traceId MDC entry used in every log line of step execution) was null for all async messages — exactly the messages that are hardest to trace. Fix: the emitter assigns a random UUID when the id is absent, preserving any caller-provided id.

Recommendations (not implemented)

9. Dead-letter strategy · High RECOMMENDATION

A message that deserializes fine but whose processing always fails is still nack-requeued immediately and forever — the same hot loop as finding 4, just for logic errors instead of parse errors. And with finding 4's fix, unreadable messages are dropped outright. Both belong in a dead-letter queue. Suggested follow-up: declare the work queues with a x-dead-letter-exchange argument and route rejected messages to a queue.<Service>.dlq; bound the requeue count by inspecting the x-death header and rejecting with requeue=false past a threshold. Caution: adding queue arguments to an existing queue makes queueDeclare fail (406 precondition); existing queues must be migrated or a policy-based DLX used instead (broker-side policy avoids the redeclare problem entirely and is the recommended route).

10. Fixed 5-second retry delay · Low RECOMMENDATION

The start-retry loop now runs indefinitely at a fixed 5 s interval. That is acceptable (one lightweight connect attempt per 5 s), but exponential backoff with a cap (e.g. 5 s → 60 s) would reduce log noise and broker pressure during long outages.

11. Implicit RabbitMQ client configuration · Low RECOMMENDATION

application.yaml configures only host/port/credentials; heartbeat, connection timeout and the network recovery interval all ride on defaults. For a long-running service these values are load-bearing — pin them explicitly in configuration (see the quarkus.rabbitmqclient.* properties of the Quarkiverse extension) so behavior is documented and survives client-library default changes.

12. Recovery listeners registered twice · Low RECOMMENDATION

registerListeners() adds the same recovery listener to both the connection and the channel, so unavailable()/recovered() run twice per event. The transitions are idempotent, so this is only duplicate log noise — registering on the connection alone would suffice.

What was not broken

  • Consumer-side at-least-once semantics are correct: manual ack after successful processing, nack-requeue on failure, basicQos(1) to bound in-flight work.
  • Queues are declared durable and messages published persistent (deliveryMode=2).
  • Application-initiated shutdowns are correctly filtered out of the failure listeners (isInitiatedByApplication).
  • Lock ordering between the service and receiver monitors is consistent (service → receiver); the callback paths take only the receiver lock and touch the service through lock-free state, so no deadlock.
  • The state machine (BatchServiceState) and its REST exposure (/status, /healtz, /readyz with 503 until started) give operators a truthful health signal — after these fixes, since previously the service could report starting forever while doing nothing.

Verification

All changes compile and the full reactor builds: mvn clean install -Dmaven.test.skip=true — BUILD SUCCESS. The repository has no test suite, so the failure scenarios above were verified by code-path analysis only. Recommended manual verification with a local broker:

  1. Start the service with RabbitMQ down → /readyz returns 503; start the broker after a few minutes → service transitions to started and consumes (finding 1).
  2. With the service running, delete queue.BatchAService in the management UI → consumer is cancelled, queue is redeclared and consumption resumes within ~5 s (findings 2/3).
  3. Publish a garbage (non-JSON) body to the queue → message is rejected once, not redelivered in a loop (finding 4).
  4. Restart the broker while POSTing to /batch-a/messages → in-flight emit either succeeds after one reconnect retry or returns an error; no message is silently lost (finding 5).
